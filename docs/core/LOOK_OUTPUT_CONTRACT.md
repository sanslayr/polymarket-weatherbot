# /look Output Contract (current)

Last updated: 2026-03-09

## 1) Scope
- Commands: `/look`
- Channel: Telegram（群聊/私聊一致）

## 2) Hard output rules
1. **单条最终消息**
2. 结构顺序固定：
   1) 环流形势对最高温影响
   2) METAR 实况
   3) 最高温区间
   4) 关注变量
   5) Polymarket
   6) 链接
3. 口径：先结论后证据；人话优先；去机械模板

## 3) Header rules
- 标题：`📍 ICAO (City) | lat/lon (+地形标签)`
- 时间：UTC + Local
- 头部风险提示固定文案（加粗）：`**🦞龙虾学习中，不提供交易建议🦞**`
- `near_window/in_window` 可启用 compact header（隐藏噪音行）

## 4) Synoptic rules
- 主导系统 + 落地影响必须给“方向性结论”。
- 若识别到静稳/边界层主导天气，`环流形势对最高温影响` 段必须显式给出：
  - `主导机制`
  - 结论应明确区分：`低云清除 / 静稳约束 / 混合加深 / 低层输送 / 背景环流与输送`
- 850hPa 平流 object 不能直接翻译成“站点主导输送”：
  - 必须先通过 representativeness / landing review
  - 至少审查：距离、窗口相位、925 耦合、站点地形代表性
  - 输出层级必须区分：`dominant / influence / background / low_representativeness`
  - 若 review 结果是 `background` 或 `low_representativeness`：
    - 正文不应继续写成“暖平流/冷平流主导”
    - 只允许作为背景约束或后段风险出现
- 对 Paris/London/Seattle 等低云雾/弱风/高湿场景，不允许只给 500/850 模板而缺少“边界层清除主导”结论。
- 主导系统行应尽量在一句话内带出“系统性质”（如偏暖锋/偏冷锋/冷暖交汇，或冷暖输送属性），不强制拆独立小节。
- 关键证据按“有信号才显示”：
  - 850 通常保留
  - 700/500 仅在有判别力时展示
- 过滤泛化 boilerplate 句
- `探空层结` / `层结-模式剖面` 规则：
  - 若有实测探空，正文优先输出 `探空层结`
  - 若无实测探空，必须回退到 `层结/模式剖面`，不能让 sounding 维度整体缺席
  - 正文至少回答其中 2 项：
    - 低层是否有浅稳层/弱逆温
    - 925–850 混合是偏强还是偏弱
    - 700hPa 附近偏干还是偏湿
    - Tmax 更偏低云清除节奏，还是低层输送/背景环流与输送
- 若 `主导机制` 判到 `static_stable / boundary_layer_clearing / mixing_depth`：
  - 850 review 默认只能做次级证据，除非其 `surface_role == dominant`
  - 不允许远端或低代表性的 850 输送信号盖过低层湿稳/清除节奏

## 5) METAR block rules
- 顶部保留“今日已观测最高温（时间）”。
- 最新报标题内显示上一报时间。
- 风：需要写与上一报相比的变化（转向/风速增减）。
- 云：写全层结构，并给上一报全层对比（新增/消退/重排/稳定）。
- 天气现象：保留原码并尽量给中文释义（例：`-RA（小雨）`）。
- 短时趋势语句应优先匹配站点采样节律（半小时/整点），若出现 SPECI 加密采样可缩短判读窗口；若识别到 SPECI 触发前兆，也应降低“单报定局”强度。
- 在稳定天气（无SPECI活跃/前兆）下，关注变量不主动展开 SPECI 讨论，聚焦下一关键报的温度/风场/云量情形。
- 对常规每小时一报站点，在近/窗内/窗后关键阶段应优先提示“下一关键报时间”（如 `xx:53`），用于判定是否封顶。
- 若识别 `boundary_layer_clearing / static_stable / mixing_depth`：
  - `关注变量` 需优先盯云底、温露差、风速、开云节奏
  - 不应继续只给“风向变化/斜率变化”的通用模板

## 6) Tmax interval rules
- 标题固定：`可能最高温区间（仅供参考）`
- 形式：`主带区间 + 条件尾部`（合并在同一行表达）
- 在进入区间计算前，必须先完成一次 `resolved_window` 解析：
  - 若模型主窗与实时观测明显冲突，区间、相位、关注变量一律使用重锚后的窗口
  - 不允许“文案按实况说已横盘/已到峰值时段，但区间仍沿用模型晚段尾部”
- 在窗口仲裁前，必须先完成一次 `temperature_shape_analysis`：
  - `multi_peak_state` 只用于讨论“分离式二峰/多峰”
  - `plateau_state` / `observed.plateau_state` 用于讨论“峰值附近宽平台/持续守高”
  - `future_candidate` 必须区分：
    - `primary_remaining_peak`：仍在等待主峰到来
    - `secondary_peak_candidate`：存在结构性次峰候选
  - 不允许把“峰值附近宽平台”直接写成“独立二峰”
- 约束优先级：
  - 低云持续压制
  - 降水/雨后残余冷却
  - 量化 METAR 台阶风险
  - 午后平台期太阳辐射边际下降（结合太阳几何 + METAR 多层云量/天气现象 `radiation_eff`）
  - 窗口结束后惯性上冲抑制（时钟已过窗口尾且斜率未放大时压制上沿）
  - 夜间增温复核（暖平流 + 混合/云被/露点/气压组合证据出现时，仅保留“小幅回升”空间）
  - post-window 已观测高点守高逻辑
- 晴空识别需将 `SKC` 与 `CLR/CAVOK` 同等视作 clear-sky（用于辐射与上沿约束判据）。
- `post` 阶段应避免“文案说已定、区间却给大新高尾巴”的冲突。
- `far` 且实况-模式偏差大时，应主动放宽区间（体现发散可能），避免给出过窄假精度。

## 7) Post-window messaging rules
- 实时温度口径必须先拆成两层：
  - `short_term_state`：短线再冲/转弱
  - `daily_peak_state`：全天高点是否允许上升到“已锁定”表述
- 不允许把“短线动能转弱”直接翻译成“全天封顶”。
- 若全天前段已出高点：
  - 环流形势段应切到“潜在反超窗口”（而非继续锚定已发生的一峰窗口）
  - 关注变量优先回答“是否还能反超前高”
  - 应给“还差多少°C”的门槛语句
  - 同时给支持/不支持反超的条件（云、雨、斜率、风场）
  - 风场变量应尽量给出“具体方向+风速阈值”场景（而非仅“转向/增风”泛化描述）
  - 区间不得低于已观测日最高温（物理一致性）
- 若站点仍明显早于常见白天峰值窗，且存在晚峰/开窗反弹倾向（如 `late_peak_share`、`cloud_break_day_share` 偏高）：
  - 可写“早峰已出现、短线动能转弱”
  - 但必须保留“全天是否锁定仍待午前后云量/降水/风场确认”的空间
  - `关注变量` 标题可用“早峰后观察”，避免直接写成“窗口后”并给出已定局语气
- 凌晨/清晨高点若与预计白天主峰差距过大：
  - 应先按 `overnight_carryover_high` 处理
  - 不应直接写成“早峰已出现”或“分离式二峰”
  - 这种场景默认仍按“白天主峰尚未到来”理解，除非 forecast/实况另有明确多峰证据
- 若 `rebound_mode == second_peak`：
  - 文案可写“弱二峰/次峰/分离式多峰”
  - 需明确触发条件（开云、降水退出、正斜率恢复、风场重排）
- 默认先写 `dominant_shape`，如：
  - `单峰尾段`
  - `峰值附近平台`
  - `平台内回摸前高`
  - `尾段窄幅震荡`
- 若 `should_discuss_second_peak == false`：
  - 不应主动写“不是独立二峰 / 不宜按二峰理解 / 不要解读成二峰”
  - 平台/回摸前高类主形态默认仅作内部 contract，不必额外展开成正文说明
- 若 `rebound_mode == retest` 或 `observed.plateau_state != none`：
  - 若无真实二峰争议，默认不单独起句讨论该形态
  - 不应把后段每次小幅回升都上升为二峰语言
- 若判定“峰值窗已过且再上冲证据弱”（settled-post）：
  - `最高温区间`改为“已观测最高温锚定”的紧凑范围
  - `关注变量`可收敛为单条结论，避免过度分支讨论
- 若触发 `obs_plateau_reanchor`（已到常见峰值时段、实况横盘守高、模型主峰却拖到明显更晚）：
  - `最高温区间`必须围绕已观测高点附近收敛
  - `峰值窗`显示为重锚后的近实时窗口，而不是原模型晚段长尾
  - 可附简短注释说明“已按实况横盘重锚峰值窗”

## 8) Market rules
- 市场日期按站点 local date 选取。
- 若 Polymarket 盘口不可用（未拉到事件/无可交易梯度/请求失败），整个 Polymarket 板块（含提醒文案）不写入报告。
- 档位解析必须兼容 F/C 与范围桶（如 `42-43F`），并兼容负号编码（`neg9` / `neg-9`）；范围桶按中点法处理（如 `54-55F` 视为中心 54.5F、覆盖约 53.5~55.5F）。
- 标签规则 forecast-first：
  - `最有可能`：天气一致性优先
  - `潜在Alpha`：需满足最小天气一致性门槛
- 远离窗口且实况-模式偏差大时（低可信阶段），应抑制 `最有可能/潜在Alpha` 标签，避免给出伪确定性。
- 在窗口后已收敛/关键小时报平稳时，默认抑制 `潜在Alpha`；但若存在明确再冲高证据（如SPECI活跃/前兆、夜间回升证据、斜率再放大+暖平流落地），可恢复 `潜在Alpha`。
- 若出现 `最有可能/潜在Alpha` 等强调标签，标题下需追加轻量风险提醒（可幽默语气），明确“非投资建议”。
- 盘口列表采用“连续主段”展示：默认覆盖【气象可能最高温区间】与【市场定价期望范围】的并集区间，并保持档位连续；不应过度压缩导致主重心档位缺失。
- 若“市场定价期望”范围明显超出当前展示档位，需自动补尾部权重档，并强制包含市场最高概率档位，避免期望区间与列表观感不一致。
- 若已有当日实况最高温，盘口中“上边界仍低于该实况最高温”的档位直接忽略（范围档按区间边界判定，不按中心值），不再参与展示与期望计算。
- 若检测到市场已结算（Resolved，且存在显式 `Yes` 结果近1.0的胜出档位），Polymarket 板块改为单句输出：`🔒 已结算：<胜出档位名>`，不再展示市场定价期望与盘口梯度。
- 常态化输出：`市场定价期望：X｜L~H（N%范围）`；N 为动态覆盖比例（按盘口形态自适应），默认按 Bid/Ask mid 权重估计（边缘占比高时给出参考性提示）。
- 排版层次：Polymarket 板块内先 `**市场定价期望**`，后 `**博弈区间**`，两者均作为同级 heading 展示；两标题之间及 `博弈区间` 与档位列表之间不插空行。
- 当盘口分布重心与气象主看区间偏离明显时，可选输出偏离提示（当前默认不额外提示，仅展示隐含期望/主带）。

## 9) Unit rules
- 美国站点（ICAO `K*`）：所有温度相关展示统一用 °F（不再附带 °C），包括 METAR 温度/露点、同小时温度偏差、已观测最高温、Tmax 区间、Polymarket 期望与区间。
- 美国站点在“已观测最高温锚定”时按 ±0.5°F 量化区间处理（内部转换为 °C 区间用于区间锚定/物理约束；盘口过滤仍按已观测真值）。
- 其余站点：默认显示 °C；METAR 实况温度、已观测最高温及“较上一报变化”按原始精度展示（无小数不补 `.0`，有小数则保留）；同小时温度偏差固定保留 1 位小数。
- 非美国站点在“已观测最高温锚定”时，若 METAR 为整数°C，内部按量化区间处理（如 `-5°C` 视作约 `[-5.5, -4.51]°C`），避免伪精度；盘口过滤仍按已观测真值。

## 10) Implementation guardrail（实现护栏，强约束）
- 涉及窗口时差/相位时差的实现，统一使用 `_hours_between_iso(...)`。
- 不允许新增分散 tz 差分逻辑（手写 `fromisoformat` 差分 / 手动 `replace(tzinfo=...)` 差分）。
- 若必须例外，需在代码写 `TZ-WAIVER` 注释并同步 `TECHNICAL_IMPLEMENTATION_NOTES.md` 记录。

## 11) Layout / typography rules（排版总则）
- 标题层级：主段落用 `**...**`；同级段落避免混用 bullet 与 heading。
- `METAR` 内：
  - `• 今日已观测最高温`
  - `**最新报：...**（上一报 ...）`
  - 后续字段维持 `• **字段**：值` 的平铺样式。
  - 对“峰值窗已过/关键小时报平稳”等已发生事实，应收敛到独立 `**实况分析**` 小块（不放进关注变量）。
- Polymarket 内：
  - `**市场定价期望**` 在前，`**博弈区间**` 在后。
  - 期望值行用缩进箭头：`↳ X｜L~H（N%范围）`（箭头前不再加 bullet）。
  - `市场定价期望` 与 `博弈区间` 之间不空行；`博弈区间` 与档位列表之间不空行。
- 句尾风格：关键数值行默认不强制句号，减少视觉噪声。
- 底部仅保留 `🔗` 链接行；不再额外输出“龙虾带单”提示语。
