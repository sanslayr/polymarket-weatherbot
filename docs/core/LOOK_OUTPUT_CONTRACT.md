# /look Output Contract (current)

Last updated: 2026-03-02

## 1) Scope
- Commands: `/look`, `/lookhelp`
- Channel: Telegram（群聊/私聊一致）

## 2) Hard output rules
1. **单条最终消息**（禁止预告/占位）
2. 结构顺序固定：
   1) 环流形势对最高温影响
   2) METAR 实况
   3) 最高温区间
   4) 关注变量
   5) Polymarket
   6) 链接
3. 口径：先结论后证据；人话优先；去机械模板

## 3) Header rules
- 标题：`📍 ICAO (City) | lat/lon (+地形标签)`
- 时间：UTC + Local
- 头部风险提示固定文案（加粗）：`**🦞龙虾学习中，不提供交易建议🦞**`
- `near_window/in_window` 可启用 compact header（隐藏噪音行）

## 4) Synoptic rules
- 主导系统 + 落地影响必须给“方向性结论”。
- 主导系统行应尽量在一句话内带出“系统性质”（如偏暖锋/偏冷锋/冷暖交汇，或冷暖输送属性），不强制拆独立小节。
- 关键证据按“有信号才显示”：
  - 850 通常保留
  - 700/500 仅在有判别力时展示
  - 过滤泛化 boilerplate 句

## 5) METAR block rules
- 顶部保留“今日已观测最高温（时间）”。
- 最新报标题内显示上一报时间。
- 风：需要写与上一报相比的变化（转向/风速增减）。
- 云：写全层结构，并给上一报全层对比（新增/消退/重排/稳定）。
- 天气现象：保留原码并尽量给中文释义（例：`-RA（小雨）`）。

## 6) Tmax interval rules
- 标题固定：`可能最高温区间（仅供参考）`
- 形式：`主带区间 + 条件尾部`（合并在同一行表达）
- 约束优先级：
  - 低云持续压制
  - 降水/雨后残余冷却
  - 量化 METAR 台阶风险
  - 午后平台期太阳辐射边际下降（结合太阳几何 + METAR 多层云量/天气现象 `radiation_eff`）
  - 窗口结束后惯性上冲抑制（时钟已过窗口尾且斜率未放大时压制上沿）
  - 夜间增温复核（暖平流 + 混合/云被/露点/气压组合证据出现时，仅保留“小幅回升”空间）
  - post-window 已观测高点守高逻辑
- 晴空识别需将 `SKC` 与 `CLR/CAVOK` 同等视作 clear-sky（用于辐射与上沿约束判据）。
- `post` 阶段应避免“文案说已定、区间却给大新高尾巴”的冲突。
- `far` 且实况-模式偏差大时，应主动放宽区间（体现发散可能），避免给出过窄假精度。

## 7) Post-window messaging rules
- 若全天前段已出高点：
  - 环流形势段应切到“潜在反超窗口”（而非继续锚定已发生的一峰窗口）
  - 关注变量优先回答“是否还能反超前高”
  - 应给“还差多少°C”的门槛语句
  - 同时给支持/不支持反超的条件（云、雨、斜率、风场）
  - 风场变量应尽量给出“具体方向+风速阈值”场景（而非仅“转向/增风”泛化描述）
  - 区间不得低于已观测日最高温（物理一致性）
- 若判定“峰值窗已过且再上冲证据弱”（settled-post）：
  - `最高温区间`改为“已观测最高温锚定”的紧凑范围
  - `关注变量`可收敛为单条结论，避免过度分支讨论

## 8) Market rules
- 市场日期按站点 local date 选取。
- 若 Polymarket 盘口不可用（未拉到事件/无可交易梯度/请求失败），整个 Polymarket 板块（含提醒文案）不写入报告。
- 档位解析必须兼容 F/C 与范围桶（如 `42-43F`），并兼容负号编码（`neg9` / `neg-9`）；范围桶按中点法处理（如 `54-55F` 视为中心 54.5F、覆盖约 53.5~55.5F）。
- 标签规则 forecast-first：
  - `最有可能`：天气一致性优先
  - `潜在Alpha`：需满足最小天气一致性门槛
- 远离窗口且实况-模式偏差大时（低可信阶段），应抑制 `最有可能/潜在Alpha` 标签，避免给出伪确定性。
- 若出现 `最有可能/潜在Alpha` 等强调标签，标题下需追加轻量风险提醒（可幽默语气），明确“非投资建议”。
- 盘口列表采用“连续主段”展示：覆盖气象主看区间 + 市场主带（mid加权）；若主带下沿贴近下一档，会补一档下沿连续位；高温尾部权重不低时补一个上边缘档位（如 13°C+），避免列表过长。
- 若已有当日实况最高温，盘口中“上边界仍低于该实况最高温”的档位直接忽略（范围档按区间边界判定，不按中心值），不再参与展示与期望计算。
- 若检测到市场已结算（Resolved，且存在显式 `Yes` 结果近1.0的胜出档位），Polymarket 板块改为单句输出：`🔒 已结算：<胜出档位名>`，不再展示市场定价期望与盘口梯度。
- 常态化输出：`市场定价期望：X｜L~H（N%范围）`；N 为动态覆盖比例（按盘口形态自适应），默认按 Bid/Ask mid 权重估计（边缘占比高时给出参考性提示）。
- 排版层次：Polymarket 板块内先 `**市场定价期望**`，后 `**博弈区间**`，两者均作为同级 heading 展示；两标题之间及 `博弈区间` 与档位列表之间不插空行。
- 当盘口分布重心与气象主看区间偏离明显时，可选输出偏离提示（当前默认不额外提示，仅展示隐含期望/主带）。

## 9) Unit rules
- 美国站点（ICAO `K*`）：所有温度相关展示统一用 °F（不再附带 °C），包括 METAR 温度/露点、同小时温度偏差、已观测最高温、Tmax 区间、Polymarket 期望与区间。
- 美国站点在“已观测最高温锚定”时按 ±0.5°F 量化区间处理（内部转换为 °C 区间用于区间锚定/物理约束；盘口过滤仍按已观测真值）。
- 其余站点：默认显示 °C；METAR 实况温度、已观测最高温及“较上一报变化”按原始精度展示（无小数不补 `.0`，有小数则保留）；同小时温度偏差固定保留 1 位小数。
- 非美国站点在“已观测最高温锚定”时，若 METAR 为整数°C，内部按量化区间处理（如 `-5°C` 视作约 `[-5.5, -4.51]°C`），避免伪精度；盘口过滤仍按已观测真值。

## 10) Layout / typography rules（排版总则）
- 标题层级：主段落用 `**...**`；同级段落避免混用 bullet 与 heading。
- `METAR` 内：
  - `• 今日已观测最高温`
  - `**最新报：...**（上一报 ...）`
  - 后续字段维持 `• **字段**：值` 的平铺样式。
- Polymarket 内：
  - `**市场定价期望**` 在前，`**博弈区间**` 在后。
  - 期望值行用缩进箭头：`↳ X｜L~H（N%范围）`（箭头前不再加 bullet）。
  - `市场定价期望` 与 `博弈区间` 之间不空行；`博弈区间` 与档位列表之间不空行。
- 句尾风格：关键数值行默认不强制句号，减少视觉噪声。
- 底部仅保留 `🔗` 链接行；不再额外输出“龙虾带单”提示语。
