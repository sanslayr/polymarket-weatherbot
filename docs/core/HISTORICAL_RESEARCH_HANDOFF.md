# 历史数据获取与分析调研对接文档（Weatherbot）

Last updated: 2026-03-09

## 1. 文档目的

本文件用于向后续 ChatGPT 调研/策划任务提供**精确上下文与约束**，确保输出能直接对接当前 `polymarket-weatherbot` 的工程状态，而不是泛化方案。

---

## 2. 当前项目在做什么（必须对齐）

我们当前线上核心能力是 `/look`：

- 输入：站点（主要为 METAR 站点）
- 输出：
  - 最高温可能区间（Tmax interval）
  - 基于预报+实况的环流背景解释
  - Polymarket 盘口展示（展示层）

当前策略主线：

1. **先稳定天气区间判断**
2. **再独立设定市场标签阈值**
3. 严禁两者互相污染（天气结论不可被盘口反向牵引）

---

## 3. 现有代码分层（调研方案必须按此接入）

- `scripts/hourly_data_service.py`  
  小时预报抓取、缓存、窗口识别

- `scripts/metar_analysis_service.py`  
  METAR/SPECI 实况特征提取与诊断

- `scripts/analysis_snapshot_service.py`  
  结构化分析快照主入口（当前 report 的主要分析 handoff）

- `scripts/peak_range_service.py`  
  最高温区间分析与结构化 summary/block 产出

- `scripts/market_label_policy.py`  
  市场标签门控与阈值策略（已独立）

- `scripts/polymarket_render_service.py`  
  盘口解析与展示渲染

- `scripts/report_render_service.py`  
  报告段落编排与整合

- `scripts/telegram_report_cli.py`  
  仅做入口编排与最终文本输出（不承载业务规则）

---

## 4. 已确认的关键事实（调研必须基于这些事实）

1. 已完成“天气判断 vs 市场展示”解耦（单向依赖）。
2. 旧概率计算层已退役，不再作为演进方向。
3. 当前痛点是：**窗口期外 Tmax 偏差较大**。
4. 观察到“站点规律稳定”，但稳定性是**天气型条件稳定**，不是每天同一模式。
5. 当前样本积累还不够，必须有小样本可运行方案。
6. `/look` 执行链路需保持轻量与响应速度，不接受显著变慢的大方案。
7. 历史训练主过程不应直接并入 runtime repo；推荐 `research repo -> artifact -> runtime repo` 的连接方式。

## 5. 要调研并回答的核心问题

### 5.1 历史学习架构

请给出一个可增量迭代的架构：

- 数据层（原始、特征、标签）
- 校正层（bias / quantile）
- 评估层（按站点、按天气型、按窗口阶段）
- 上线层（在线推理与回退策略）

要求：支持先小步上线，再逐步增强。

### 5.2 数据清单与来源

请给出字段级清单，并说明：

- 来源（METAR/预报快照/ERA5/周边站）
- 更新频率
- 存储粒度
- 预计数据量（单站、多站）
- 获取难度与工程复杂度

### 5.3 小样本可用策略

请重点设计：

- `ΔT = obs_tmax - raw_forecast_tmax` 的校正路线
- 分层回退链：`station×regime -> station -> regime -> global`
- 最小样本门槛建议
- 样本不足时的不确定性放宽策略（区间扩展而非硬预测）

### 5.4 预报快照回放

请给出“当时可见信息”的可追溯方案，防止训练/回测信息泄漏：

- 快照存储结构建议（站点/日期/时次/provider）
- 回放读取机制
- 版本化与可复现实验建议

### 5.5 ERA5 可行性（重点）

请判断 ERA5 是否适合作为环流重分析历史数据底座，并给出：

- 推荐变量与层次（满足 Tmax 校正所需最小集合）
- 时空分辨率取舍
- 下载接口与抓取方式
- 数据量估算与成本控制方案
- 与现有流程的最小侵入接入点

## 6. 方法论约束（避免方向跑偏）

1. 不做“纯站点固定偏差”  
   应做 `station × regime` 条件偏差。

2. 不做“仅环流定量替代历史校正”  
   应该是：历史校正主干 + 环流增量修正。

3. 优先区间校准而非点值炫技  
   建议输出分位区间（如 P20/P50/P80）。

4. 样本不足时，宁可保守扩大区间，不做激进偏置修正。

---

## 7. 交付物要求（给 ChatGPT 的输出格式）

请按以下结构输出：

### A) 架构方案
- 模块图（文字版）
- 数据流与调用顺序
- 与现有代码模块的对接关系

### B) 数据方案
- 字段清单（原始/特征/标签）
- 每项数据的来源、频率、规模、获取难度
- 最小可用数据集 vs 增强数据集

### C) 实施计划
- `v0`（2~4周）：小样本可上线版本
- `v1`（后续）：引入更完整 regime 与 ERA5 增强
- 每阶段里程碑与验收条件

### D) 评估体系
- 基础指标：MAE、区间覆盖率、分位数校准误差
- 分组评估：按站点、regime、窗口阶段
- 回测与线上一致性检查

### E) 风险与缓解
- 数据缺失/时效/质量风险
- 工程复杂度与性能风险
- 模型漂移与季节切换风险

## 8. 工程接入硬约束（必须遵守）

1. 不恢复旧概率层。  
2. 不把新业务规则继续堆入 `telegram_report_cli.py`。  
3. 天气判断链路不读取盘口状态。  
4. 市场标签策略不反向改写天气区间。  
5. 新增能力优先作为“可开关模块”接入，便于 A/B 与回滚。  
6. 历史训练、ERA5 拉取、回测与 notebook 优先放在独立 research repo，而不是直接压进当前 skill repo。  

---

## 8.1 推荐连接方式

建议采用双仓边界：

- `weatherbot-runtime`
  - 当前 skill repo
  - 只负责在线推理、报告、市场监控与后续执行
- `weatherbot-research`
  - 负责历史数据、ERA5、样本构建、训练、回测、评估

runtime 仅通过 artifact 接入 research 结果，例如：

- station priors
- analog index
- regime priors / embeddings
- posterior weights
- calibration tables
- manifest / schema version

## 9. 建议的最小接入路线（供调研参考）

### 阶段 v0（小样本）

- 新增一个轻量校正层（例如 `bias_adjust_v0`）：
  - 输入：当前 `analysis_snapshot` / `peak_range_service` 产物 + 近期实况特征
  - 输出：校正后区间（含不确定性）
- 只启用 `global + station` 两层偏差；
- `regime` 先用于“区间放宽/收窄门控”，暂不做细颗粒拟合。

### 阶段 v1（样本扩展后）

- 启用 `station×regime` 条件偏差学习；
- 引入 ERA5 环流特征增强；
- 加入周边站一致性约束，降低单站异常扰动。

---

## 10. 一句话目标

在不牺牲 `/look` 速度与模块清晰度的前提下，构建“可持续学习”的 Tmax 区间校正体系：  
**先稳，再准；先可用，再精细。**
